# تقرير الاختبار النهائي

## نطاق الاختبار

تم اختبار النسخة النهائية بوصفها مشروع Front-end ثابتًا يعمل بـ HTML وCSS وJavaScript و`localStorage`، دون افتراض وجود خادم خلفي أو ذكاء اصطناعي أو دفع حقيقي.

## الاختبارات المنفذة والنتائج

1. **سلامة JavaScript**
   - فحص جميع ملفات JavaScript التسعة عبر `node --check`.
   - النتيجة: ناجح دون أخطاء تركيبية.

2. **سلامة HTML والوصول الأساسي**
   - فحص 10 صفحات HTML.
   - التأكد من `lang="ar"` و`dir="rtl"` ووسم viewport والوصف والعنوان وfavicon ووجود `<main>`.
   - التأكد من عدم تكرار المعرّفات `id`.
   - التأكد من سلامة الروابط المحلية والمقاطع الداخلية.
   - التأكد من وجود تسميات صريحة لحقول النماذج.
   - فحص 62 زرًا والتأكد من تحديد `type` لكل زر.
   - النتيجة: ناجح دون تحذيرات أو روابط ميتة.

3. **سلامة CSS**
   - فحص توازن الأقواس في ملف التنسيق: 441 قوس فتح مقابل 441 قوس إغلاق.
   - النتيجة: ناجح.

4. **اختبار منطق البيانات والتشغيل**
   - تهيئة إصدار البيانات التجريبية رقم 3.
   - التحقق من ربط جميع الإحالات التجريبية بمعرّف ورشة.
   - مطابقة ثلاث ورش في المدينة نفسها.
   - منع تكرار الورشة داخل الطلب.
   - إنشاء الإحالة الثانية والثالثة داخل رقم الطلب نفسه.
   - منع الإحالة الرابعة وفق الحد الأقصى.
   - إنشاء رسالة واتساب تتضمن رقم الطلب وبيانات السيارة.
   - استخدام رابط مشاركة واتساب آمن للبيانات التجريبية بدل توجيه الرسالة إلى رقم وهمي.
   - تحديث حالة الإحالة عند فتح واتساب.
   - تسجيل الرسم مرة واحدة فقط لكل إحالة ومنع التكرار.
   - النتيجة: ناجح.

5. **اختبار التقديم عبر خادم ثابت**
   - تشغيل المشروع عبر خادم HTTP محلي.
   - طلب جميع الصفحات الرئيسية والملفات الأساسية، ومنها CSS وJavaScript وfavicon.
   - النتيجة: جميعها أعادت حالة HTTP 200 ومحتوى غير فارغ.

6. **فحص توافق النطاق**
   - لا توجد أسعار إصلاح متوقعة أو نطاقات تكلفة.
   - لا يوجد تشخيص نهائي.
   - تظهر ورشة واحدة فقط في كل إحالة.
   - خيار عدم الاتفاق على السعر يظهر كسبب لطلب بديل، وليس كتقدير سعري من المنصة.

## سيناريوهات القبول الجاهزة للتجربة

1. إنشاء طلب صيانة دورية حتى تسجيل الإحالة وعرض ورشة واحدة.
2. إنشاء طلب مشكلة، والإجابة عن الأسئلة، ثم مشاهدة التوجيه المبدئي.
3. فتح واتساب أو نسخ الرسالة الجاهزة.
4. طلب ورشة ثانية وثالثة والتأكد من ثبات رقم الطلب وعدم تكرار الورش.
5. استرجاع الطلب من صفحة المتابعة باستخدام الرقم والجوال.
6. تأكيد الوصول وإتمام الخدمة ثم إرسال تقييم موثق.
7. دخول الورشة وتحديث حالة إحالة ومراجعة كشف الحساب.
8. تنفيذ الدفع التجريبي وفتح الإيصال وطباعته.
9. تقديم طلب انضمام متعدد الخطوات ومتابعة حالته.

## قيد بيئة الاختبار

تعذر تشغيل متصفح رسومي آلي داخل بيئة الإنشاء بسبب سياسة حظر المتصفح المحلية. لذلك لم يُدّعَ تنفيذ اختبار بصري آلي بالنقر الفعلي. تم التعويض باختبارات البنية والروابط والخادم ومنطق التخزين والإحالات والرسوم. يوصى بإجراء جولة قبول بصرية أخيرة على Chrome أو Safari بعد النشر، خصوصًا على هاتف فعلي.
